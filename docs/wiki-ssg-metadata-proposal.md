# Wiki → SSG metadata proposal

**Status:** proposal (not yet scheduled). Authored to capture the gaps found while
adding SEO metadata (per-page titles/descriptions, sitemap, Open Graph) to the
static-site generator [`mdbook`](https://github.com/igorboss/mdbook), which renders
the `wiki-ssg` export.

## Context

The TermX Wiki publishes a space to GitHub via the `wiki-ssg` export, and a static
site is built from it by mdbook (VitePress-based, same markdown engine as the wiki).
The export contract is owned by the **backend**, not the frontend:

- Export handler: `wiki/src/main/java/org/termx/wiki/SpaceGithubDataWikiSsgHandler.java`
  (records `SsgSpaceIndex`, `SpaceGithubPage`, `SpaceGithubPageContent`).
- Models: `termx-api/.../wiki/page/{Page,PageContent}.java`, `.../sys/space/Space.java`
  (+ TS mirrors in `termx-web/app/src/app/wiki/_lib/page/models/` and
  `sys/_lib/space/model/space.ts`).
- The frontend only models pages/spaces and triggers the generic GitHub push; it does
  not emit the export files.

The export currently emits:

| Path | Content |
|---|---|
| `space.json` | `{ web, code, names }` — `web` = server `termx.web-url`, `names` = localized space names |
| `pages.json` | page tree: `SpaceGithubPage(code, contents[], children[])`, `SpaceGithubPageContent(name, slug, lang, contentType, modifiedAt)` |
| `pages/<slug>.md` | `PageContent.content` **verbatim** |
| `attachments/<pageId>/<file>` | attachment bytes |
| `resources/<type>/<name>.json` | embedded FHIR resources |

## Gaps (exists vs missing)

| Capability | Today |
|---|---|
| Page markdown body | exported verbatim |
| Page title | only in `pages.json` as `name` — **never an H1 in the `.md`** |
| Per-page description / keywords | **missing** — no model field (runtime `<meta>` uses `content.slice(0,1000)`) |
| Space localized names → `space.json` | exists (`names`) |
| Space description / site title / theme / URL | **missing** — only hand-authored in `.mdbook/config.yml` |
| Space default language / languages list | **missing** — global `environment.contentLanguages`/`defaultLanguage` only, not per-space, not exported |
| Multi-language pages | exists (one `PageContent` per lang, grouped in `pages.json`) |
| Sitemap | generated by mdbook; nothing on the wiki side |

Because these are missing, mdbook currently *infers* what it can: it injects the page
`name` as the `<title>`, derives a `<meta description>` from the first paragraph, and
auto-detects the canonical URL in CI for the sitemap/canonical/Open Graph tags. Moving
this metadata to the source removes the guessing and lets authors control it.

## Proposal

### A. Predefined H1 from the page name

Materialize the page `name` as the first `# H1` of each exported `pages/<slug>.md`.
One line in `SpaceGithubDataWikiSsgHandler` (prepend `"# " + name + "\n\n"` when the
body does not already start with an H1). Makes the `.md` self-describing and gives every
page a visible H1 — an on-page SEO signal the body lacks today.

- **Alternative (no backend change):** mdbook prepends `# {name}` when a page has no H1.
  Cheaper, but the exported `.md` stays title-less. Prefer the source-side change so the
  export is self-contained; mdbook keeps its injection as a fallback for un-re-exported repos.

### B. Per-page metadata: `description`, `keywords`

Add per-language content metadata:

- Model: `PageContent.description: String`, `PageContent.keywords: List<String>` (nullable),
  plus the TS mirror and an editor field (page setup / header).
- Export: extend `SpaceGithubPageContent` →
  `(name, slug, lang, contentType, description, keywords, modifiedAt)`.
- mdbook: read `description`/`keywords` from `pages.json` into per-page frontmatter,
  falling back to the first-paragraph heuristic when absent.

Replaces the runtime `content.slice(0,1000)` `<meta>` hack with authored, stored values.

### C. Project metadata → generator config

Add space-level fields so the generator config no longer has to be hand-authored:

- Model (`Space`): `description: LocalizedName`, `defaultLanguage: String`,
  `languages: List<String>`, optional `siteUrl: String`.
- Export: extend `SsgSpaceIndex` →
  `{ web, code, names, description, defaultLang, langs, siteUrl }`.
- mdbook: consume these as defaults for `site.description` / `site.lang` / languages
  (it already uses `names` for `site.title`). `.mdbook/config.yml` then shrinks to
  generator-only concerns (`theme`, `tx-server`, `search`).

This is the "project metadata belongs in the config" item — sourced from the wiki
instead of hand-authored in each repo.

### D. Languages

The `defaultLanguage` / `languages` from **C** replace today's global-environment
inference (`environment.contentLanguages`, `environment.defaultLanguage`). The space
owns its language set and default, and both are exported in `space.json`. mdbook uses the
explicit list instead of inferring which languages exist from `pages.json` contents.

### E. Sitemap

Handled on the generator side — mdbook now emits `sitemap.xml` + `<link rel="canonical">`
+ Open Graph tags, auto-detecting the canonical URL in CI
(`https://<owner>.github.io/<repo>/`, or a CNAME domain). The only wiki-side input is the
optional `space.siteUrl` from **C**, needed when the site is served from a custom domain
that CI can't infer.

## Compatibility & rollout

- All new fields are **additive and nullable**. Old exports keep working; mdbook already
  falls back (name → title, first paragraph → description, CI → URL) when they are absent.
- Order of value: **B** (descriptions) and **C** (project metadata) give the biggest
  authoring/SEO win; **A** is a one-liner; **D**/**E** are small once **C** lands.
- Frontend work is limited to model mirrors + editor fields; the export contract change is
  localized to `SpaceGithubDataWikiSsgHandler` and the two record types.

## Non-goals

- No change to the markdown body syntax or the wiki rendering pipeline.
- No change to the PDF/HTML `wiki-export` feature (unrelated to `wiki-ssg`).
- The generator (`mdbook`) stays external; it only gains readers for the new fields.
